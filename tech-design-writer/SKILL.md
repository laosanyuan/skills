# Tech Design Writer

通过对话把 PRD 翻译成中等深度的**技术设计文档**(开发方案),产出给 team lead 做 sprint 规划用的颗粒度 —— 架构、技术栈、数据模型、接口清单、模块拆分、风险点。

核心是三件事:**先读 PRD**;**六个维度没问清楚不动笔**;**不当全知工程师** —— 遇到方向性决策(单体 vs 微服务、自研 vs 用云服务、消息队列选哪个)永远把选项摆出来让用户拍板,不要在文档里悄悄做决定。

## 工作流概览

每次被触发时按以下顺序处理:

1. **判断是不是该用本 skill**(不该用就直接告诉用户) → 2. **读已有 PRD** → 3. **过一遍 6 个必问维度,该问的问** → 4. **检测 PRD 约束 vs 你打算选的方案有没有冲突** → 5. **写或更新技术设计文档** → 6. **告诉用户文档在哪、关键决策点、未决项**

千万不要跳过第 3 步直接动笔 —— 技术设计的每个决策都有 trade-off,**没问清楚就替用户选,等于把工程债务藏到文档里**。

## transcript.md 是什么

skill 工作流里反复让你"写到 transcript.md"——这是给用户的**决策日志**,不是给 LLM 看的废纸。统一用 4 段结构,reviewer 一眼看懂哪儿找信息:

1. **用户已明示的约束**(只写用户原话或明确推论的,不把"建议方案"放这里)
2. **我会问的问题**(按维度,每条带 a/b/c 选项 + 建议方案)
3. **兜底默认值 + 理由**(表格形式:决策点 / 默认值 / 理由)
4. **PRD 内部张力 / 待确认事项**(跟 TECH-DESIGN 文档里标注的对得上)

**结构化的目的**: reviewer 想快速知道"我说了什么、模型问了什么、模型自己定了什么"。没结构容易跳过、默认决策没被发现。

## 0. 先判断该不该用本 skill

被触发后仍要确认:

- 用户实际想**写/改 PRD 本身** → 告诉用户:"这看起来是产品需求层面的工作,prd-writer 更合适。你确实是想做技术方案吗?"
- 用户想**写具体代码 / 配 CI/CD / 写脚本** → 直接答:"这个事我直接帮你做就行,不用走 tech-design-writer 流程。"
- 请求**混合**(如"做技术方案 + 写 user 表 SQL")→ 先做技术设计部分,做完问:"SQL 那部分我接着做还是另开?"

## 1. 启动时先读已有 PRD

被触发后,先用 Glob 看当前工作目录有没有 `PRD-*.md` 文件:

- **0 个** → **不能开始**。告诉用户:"我需要先看到 PRD 才能做技术方案。当前目录没有 PRD-*.md 文件。要不要先用 prd-writer 整理出 PRD 再来?或者你已经有 PRD 但放在别的地方,告诉我路径?"
- **1 个** → Read 一遍,理解产品定位、所有 F00X 功能、已有的约束(性能/边界/排除项)
- **2 个及以上** → **不要自己猜**。问用户:"我看到目录里有几份 PRD([列文件名]),你这次要做技术方案的范围是哪一份?或者覆盖几份?" 拿到答案再决定读哪些

如果同目录已有同产品的 `TECH-DESIGN-*.md` 文件,**也读一遍**,了解之前已经定的选型,避免新设计跟旧的打架。

## 2. 六个必问维度

PRD 关心"做什么";技术设计关心"怎么做"。这中间有 6 个维度,**用户不主动说就主动问**,问的时候带选项让用户快速选。

### 2.1 架构形态

决定整体形态:单体 / 前后端分离 / 微服务 / serverless / 桌面 / 移动 App / 嵌入式。

带选项追问:
> "整体架构你倾向哪种?(a) 前后端分离的 Web 应用 (b) 微信小程序 + 后端 (c) 原生 App + 后端 (d) 桌面应用 (e) 多端齐全 (Web + 移动) (f) 其他?"

为什么必问: 这一个决策几乎决定了之后所有选型的范围。

**分期意识**: 架构选型也要分期 —— Phase 1 (MVP) 通常用单体 / 简单部署足够,**不要一上来就微服务 / K8s / 多集群**(几乎一定 over-engineering)。在 §7 风险章节注明"当 QPS > X 或团队规模 > Y 时再考虑拆服务",留好升级路径就行。

### 2.2 使用场景

谁用、在哪用、怎么进来、用量大概什么级别:

- 内部工具 vs 公网产品 vs B2B SaaS
- 国内 vs 跨国(决定时区/合规/CDN)
- 网络环境(公司内网 / 弱网 / 离线优先)
- 接入方式(直接访问 / 嵌入到其他系统 / 第三方授权)

为什么必问: 同样是"用户登录",内部 SSO 接公司域账号、和公网产品做手机号验证码,完全不是一回事。

### 2.3 技术栈

前端 / 后端 / 数据库 / 缓存 / 消息队列 / 部署平台。

**重要**: 不要替用户选。带选项问:
> "后端语言/框架你们团队倾向?(a) Node.js (Express/NestJS/Hono) (b) Python (FastAPI/Django) (c) Go (d) Java/Kotlin (Spring) (e) 其他/听你建议?"

如果用户说"听你建议",再给出 2-3 个候选 + 各自适用场景 + 推荐,**仍然让用户选**。

技术栈决策维度(每个都要问到):

- **语言/框架**(前后端各一)
- **数据库**(关系型/文档型/兼用,具体哪个)
- **缓存**(要不要,Redis 还是其他)
- **消息队列**(要不要,Kafka / RabbitMQ / Redis Streams / 云服务)
- **部署**(自建 K8s / 云函数 / Docker Compose / 直接二进制)
- **CDN / 对象存储**(图片视频要不要)

### 2.4 性能与规模

数字驱动选型。问:

- 预期 **QPS** / 日活 / MAU(给个量级即可,1k / 10k / 100k / 1M+)
- 数据增长速度(每天产生多少条 / 多少 GB)
- 是否有**峰值**(秒杀、双 11 类场景)
- 单次请求的响应时间预算(给用户的可感知延迟)

为什么必问: 1k QPS 和 100k QPS 完全不同方案;有秒杀就要做异步队列、有缓存策略、可能要读写分离。

如果用户答"还不知道,先做出来再说",**给一个合理初版预设 + 标"建议方案"**:
> 建议方案: 按 100 DAU / 100 QPS 设计初版,后续观测后再扩 (待用户确认)

### 2.5 合规 / 安全

问:

- 用户数据是否包含 **PII**(身份证、手机号、地址、生日)
- 是否需要符合 **GDPR / 等保 / SOC2** 等
- **审计日志**需求(谁什么时候做了什么)
- **数据出境**(国内服务器 only 还是全球)
- **认证授权**(自建账号 / OAuth / SSO / 多租户)

为什么必问: 这些决策影响数据库表设计、加密策略、日志方案、部署地域。早问比上线后改容易 10 倍。

### 2.6 现有系统集成

- 从零搭还是在**已有系统**上加
- 如果集成,接哪些(自家其他服务、第三方支付、地图、短信、推送、CRM、SSO 提供方)
- 数据源是新建还是复用现有数据库

为什么必问: 整合现有系统 ≠ 从零搭;前者有大量"按已有 API 接、按已有 schema 写"的约束,而后者自由度大。

### 2.7 追问的节奏

不要一次抛 6 个维度让用户填空,会变成"考试官"。**每轮聚焦 1-3 个最关键的点**,该深就深;先把架构形态 + 技术栈骨架问清楚,再问性能/合规/集成。

什么时候可以不问、用默认值: 维度内的**次要细节**(具体框架版本、Redis 版本、CI 工具选 GitLab CI vs GitHub Actions),用合理默认 + 在文档里标"建议方案",让用户在 review 文档时一眼看到再说。**核心方向决策(单体 vs 微服务、关系型 vs 文档型、自建 vs 云服务)永远要问。**

## 3. 检测 PRD 约束 vs 设计选型的冲突

PRD 里的约束(性能、边界、不包含项)可能跟你打算的设计打架。**发现就停下来问,不要悄悄取舍**。

处理步骤:

1. 引用 PRD 的具体位置(F00X、章节、原文)
2. 说明你打算的方案是什么,以及哪里冲突
3. 给用户三个方向选: (a) 改 PRD 约束 (b) 换方案 (c) 接受 trade-off 并说明

例: "PRD F002 写'登录响应 ≤ 500ms',但我打算用阿里云实人认证服务做活体校验,实测一般在 800-1500ms。这是 (a) PRD 时延约束放宽到 1500ms (b) 换成不带活体的方案(纯手机号验证码,安全性低) (c) 接受这一步超 500ms,UI 上加 loading"

## 4. 写或更新文档

**所有不明确的关键决策都问清楚之后**才动笔。

### 4.1 文件命名

- 文件名: `TECH-DESIGN-[产品名].md`,产品名转小写 + 连字符(`TECH-DESIGN-shop-app.md`)
- 保存在当前工作目录(跟 PRD 同目录)
- 一个产品一份技术设计文档;如果设计覆盖多个 PRD,在文档开头说明覆盖范围

### 4.2 文档模板

## 2. 架构选型

**整体架构**: [单体 / 前后端分离 / 微服务 / serverless ...]

## 3. 技术栈

| 层 | 选型 | 理由 |
|----|------|------|
| 前端 | React + TypeScript + Vite | 团队熟悉、生态成熟 |
| 后端 | Node.js + NestJS | TypeScript 全栈,降低跨端成本 |
| 数据库 | PostgreSQL 15 | 关系型 + JSONB 灵活,支持后续复杂查询 |
| 缓存 | Redis 7 | 会话 + 验证码 + 热点缓存 |
| 部署 | Docker + 阿里云 ECS | 团队已有运维基础 |

## 4. 数据模型

**关键实体**(不细到字段级,给字段大概):

\`\`\`mermaid
erDiagram
    User ||--o{ Order : places
    Order ||--|{ OrderItem : contains
    User {
        bigint id PK
        string phone_encrypted "明文加密存储"
        string phone_hash "SHA-256, 唯一索引用"
        string username
        timestamp created_at
    }
    Order {
        bigint id PK
        bigint user_id FK
        enum status "待支付|已支付|已发货|已收货|已取消"
        decimal total
    }
\`\`\`

**关键约定**(避免开发歧义):

- **任何 `enum` 字段必须列出所有可能值**(像上面 Order.status 那样写引号说明)。**不要写裸 `enum status`** —— 开发会回头问"是哪几个值",且没法直接写迁移脚本
- **PII 字段(手机号、身份证、银行卡)必须双列**: 一列加密存(展示用,业务上需要时解密),一列哈希(unique 索引、查找定位用)。**不要把明文 PII 直接做主键或 unique 字段**
- **金额用 `decimal(precision, scale)` 不要用 float**(float 有精度坑)

**关键索引**: User.phone_hash (unique)、Order.user_id、Order.status

**Redis key 命名约定**(必读):

- **格式**: `{业务域}:{用途}:{key}`,如 `sms:code:<phone_hash>`、`login:fail:<phone_hash>`
- **涉及 PII 的 key 必须用 hash 后的值**,不要用明文手机号 —— 否则 Redis dump / 排错时 PII 直接泄漏
- **TTL 必标**: 每条 Redis key 在文档里写清 TTL,不要用 `永久`

示例:

```
sms:code:<phone_hash>       TTL 5min      验证码本体
sms:rate:<phone_hash>       TTL 60s       60s 防抖
login:fail:<phone_hash>     TTL 15min     失败计数
lock:account:<phone_hash>   TTL 15min     锁定标记
jwt:blacklist:<jti>         TTL = token余命  退出登录吊销
```

**对应 PRD**: User ↔ F001, Order ↔ F004

## 5. 接口清单

| 编号 | 方法 | 端点 | 描述 | 鉴权 | 对应功能 |
|------|------|------|------|------|---------|
| A001 | POST | /api/auth/sms-code | 发送短信验证码 | 无 | F001 |
| A002 | POST | /api/auth/login | 校验验证码并登录 | 无 | F001 |
| A003 | GET | /api/products | 商品列表 | 无 | F002 |
| A004 | POST | /api/orders | 创建订单 | JWT | F004 |

### 5.1 统一约定(必填,前后端契约)

- **请求 / 响应格式**: JSON
- **统一响应包**: `{ "code": 0, "message": "ok", "data": {...} }`,非 0 = 业务错误
- **HTTP 状态**: 走标准语义 —— 4xx 客户端错(401 未登录、403 无权、404 不存在、429 限流)、5xx 服务端
- **鉴权 token**: JWT in `Authorization: Bearer <token>`(实现细节见 §3)
- **分页**: list 接口默认 `page=1&page_size=20`,响应里附 `total` / `has_more`
- **trace_id**: 网关层注入 `X-Trace-Id`,所有日志和错误响应都带,**问题排查必备**
- **错误码维护**: 项目仓库根目录 `docs/error-codes.md`,所有业务码集中维护(避免散落到代码各处)
- **幂等性**: 所有 `POST` 接口都标注是否幂等;不幂等的关键接口(支付、扣库存、注册)前端必须做防抖 + 后端用 `Idempotency-Key` header 拦截重复

### 5.2 本文档定什么、不定什么(角色明确)

工程师拿到这份文档应该清楚知道**哪些字段由 sprint 启动时前后端联调定下来,哪些是文档已经约定的**。**不要写"细节字段→开发同学补充"这种甩锅措辞**,要明确分工:

**本文档(§5)定的事**(契约性,不能擅自改):

- 接口存在性(有哪些端点、HTTP 方法、URL)
- 鉴权要求(无 / JWT / 等)
- 业务幂等性(POST 接口是否幂等、撞键时怎么处理)
- 对应的 PRD 功能编号

**本文档(§5)不定但要给一个 sample 的事**(格式规范,实现时按 sample 补):

- 请求体 / 响应体的具体字段名和类型
- 错误码列表

→ **必给一个完整 sample 接口**(下面 A001),让前后端在 sprint kickoff 联调时"按此格式补每个接口的字段",**结果跟踪到对应任务的 PR 描述里**或团队 API 文档工具(Apifox / Swagger / etc.)。不要在 TECH-DESIGN 文档里把所有接口字段写满 —— 一是膨胀,二是字段微调比代码 PR 改起来麻烦。

**A001 POST /api/auth/sms-code** (sample,作为接口字段补全的格式模板)

```yaml
请求体:
  phone: string, 11位手机号,/^1[3-9]\d{9}$/
  scene: enum, "register" | "login"

响应 (200):
  code: 0
  data:
    expires_in: int, 验证码有效秒数(如 300)
    next_send_in: int, 下一次可发送的秒数(如 60)

错误码:
  PHONE_INVALID (HTTP 400): 手机号格式错误
  PHONE_RATE_LIMITED (HTTP 429): 60s 内重复发送 / 1h 内超 5 次
  SCENE_INVALID (HTTP 400): scene 参数不在允许枚举
  SMS_PROVIDER_ERROR (HTTP 502): 短信通道商错(可重试)

幂等性: 否(每次都触发实际下发)
```

### 4.3 关键写作要求

**接口编号**: `A + 三位数字`(A001、A002 ...)。和 PRD 的 `F + 三位数字` 区分。

**任务编号**: `T + 两位数字`(T01、T02 ...)。

**所有选型必须有"理由"**: 一两句话就够,讲清楚为什么选这个不选别的。**不写理由的选型对 reviewer 没用** —— 他们看完不知道这是"经过权衡的决定"还是"模型随便选的"。

**鉴权方案必须二选一**(不能"无状态 JWT + 黑名单"自相矛盾):

- **方案 A (无状态 JWT)**: access token 短(如 2h) + refresh token 长(如 7d),登出靠 refresh 失效。优点: 服务端不存 token、扩容简单。**缺点**: 立刻吊销难
- **方案 B (JWT + 黑名单)**: 单 token 长有效期,登出时 jti 加入 Redis 黑名单。优点: 立刻吊销容易。**缺点**: 每次请求查 Redis,不再是真正无状态

**填 §3 技术栈表时,先选方案再列 Redis 用途**(方案 A 不写"JWT 黑名单",方案 B 写)—— 否则技术栈表跟正文打架。

**估时附加修正**(项目用了非团队主力栈时):

§6.2 任务表估时**基于一个熟悉本栈的中级工程师**。如果选型里用了团队不熟的栈(比如团队只会 Express,方案选了 NestJS),**显式写出修正系数**:

> 若团队对 NestJS 不熟,T02-T05 各 +0.5d 学习成本

或在文档头部统一注明 buffer 系数。**不要默默假设大家都熟**。

**Mermaid 图必备 + 节点标签简短**:

- 架构图(2~6 个节点的 flowchart)—— 几乎必有
- ER 图 —— 涉及表关系时
- 状态图 —— 有订单/审批等状态机时
- **节点标签 2-8 字**,**不要**把整段细节用 `<br/>` 塞进单个节点(如 `[创建订单<br/>状态=待支付<br/>启动倒计时]` ❌)。细节(状态、倒计时、文案)放在图下方的文字步骤里说明。

**避免主观词**: 不要写"性能良好"、"易扩展"、"友好" —— 这些对工程没信息量。要写"单实例 ≤ 100 QPS"、"延迟 P99 < 300ms"。

**只写讨论过的内容**: 不要为了让文档"看起来更专业"就编"灾备方案"、"国际化"、"A/B 测试支持"这种用户没提过的章节。文档反映双方讨论过的事实。

**"建议方案"统一格式**: 模糊处或用默认值的地方只用这一种格式:

```
> 建议方案: XXX (待用户确认)
```

用 markdown 引用块(`>` 开头),让标记在视觉上跳出来。文档结尾加一个"待用户确认事项总览"清单,方便用户一次性扫一遍。

### 4.4 范围控制 + 强制分期

**第一版默认只展开 P0+P1 的细节**;P2+ 只在 §6.1 路线图列名字 + 一句话,**不展开到接口和任务粒度**。否则文档几百行,工程师拿到看不完 / 不敢动手。

详细颗粒度:

- **P0(骨架)**: 2-3 个任务把项目跑起来(repo + CI + 1 个端到端接口)
- **P1(MVP 闭环)**: 3-5 个最关键功能 → §4 数据模型 / §5 接口 / §6 任务**全部展开**
- **P2(v1.0 补齐)**: §6.1 列名字 + 一句话说明,**不展开**
- **P3+**: §6.1 只列"可能做"

**P0/P1/P2 怎么判断**:

- PRD 有优先级列 → 直接沿用(PRD P0 → 设计 P1 实现,PRD P1 → 设计 P2,依此类推)
- PRD 没有 → 主动问:"PRD 里这 N 个功能要分期吗?MVP 通常 2-3 个核心 —— 哪几个必须先做?(a) F001+F002 (b) F001+F003 (c) 都先做"
- 用户说"你判断" → 给个建议 + 标"建议方案"。**绝对不要在不分期的情况下展开所有功能**

**团队规模 / 时间约束影响分期**:

- "2 人 6 周 MVP" → P1 任务总估时 ≤ 24 人日,**超过就砍功能进 P2,不要赶工**
- "10 人 3 个月" → P1 可宽松,但仍然 MVP-first(人多也不要把所有功能塞 P1,并行会拖累质量)
- 用户没说时间/人数 → 标"建议方案"给默认 (3-4 周 MVP / 2-3 工程师)

### 4.5 反向清单: 哪些细节默认不问

下面这些**不影响架构方向**的细节,**默认不问**(行业惯例 / 由实现者决定即可,无需标"建议方案"):

- 具体框架小版本号(Express 4.18 还是 4.19)
- 代码风格 / 格式化工具(Prettier / ESLint 配置)
- 测试框架细节(Jest / Vitest 哪个)
- Logger 库选哪个(winston / pino)
- HTTP client 用 axios 还是 fetch
- 环境变量管理库

**这些事就算用户答了,对架构、上线节奏的影响也很小,大概率在开发阶段会再调一次。** 在设计阶段问反而拖慢节奏。

## 6. 迭代场景

**用户在已有 TECH-DESIGN 上加新功能**(对应 PRD 新增了 F005):
- 读已有技术设计,在对应章节追加
- §4 数据模型加新实体(注意外键关系)
- §5 接口清单追加新 A00X
- §6 任务拆分追加新 T0X
- 顶部"最后更新"改今天

**用户要换某个选型**(如"数据库改用 MySQL"):
- 不要直接覆盖,先告诉用户**影响范围**:"如果改 MySQL,§4 的 JSONB 字段要改成 JSON、§5 的 search API 性能可能下降、§7 的'读写分离方案'要改写。确认改吗?"
- 用户确认后再统一改,改完列出动了哪些章节

**用户撤销某个决策**("F003 不上 K8s 了,先 Docker Compose"):
- 类似 PRD 撤销逻辑:找到对应章节,改回来,告诉用户改了什么、影响什么(比如"那 §6 里的 K8s 任务可以删了")

## 几条容易忽视的原则

**不当"全知工程师"**。 你不知道用户的团队真的熟悉 K8s 还是只是听过、不知道他们公司有没有 Redis 运维经验。**遇到方向性决策永远把选项摆出来让用户拍板**,不要在文档里悄悄做决定。

**估时是给规划用的,不是承诺**。 §6 的估时数字是 team lead 拿来排 sprint 的参考,**永远是基于"一个熟悉本栈的中级工程师"**这个假设的(模板里已经标了)。不要给用户拍胸脯说"这就是实际工时"。

**写完之后,先看一眼"待用户确认"清单是不是太长**。 超过 5 条意味着 §2 没把架构骨架问清楚就动笔了,文档基本要重写 —— 回到追问阶段,核心搞定再继续。
