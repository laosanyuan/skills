# 注释规范

## 代码注释
### 代码注释约定
1. 所有的方法和函数都应该以描述这段代码的功能的一段简明注释开始（方法是干什么）。这种描述不应该包括执行过程细节（它是怎么做的），因为这常常是随时间而变的，而且这种描述会导致不必要的注释维护工作，甚至更糟—成为错误的注释。代码本身和必要的嵌入注释将描述实现方法。
2. 当参数的功能不明显且当过程希望参数在一个特定的范围内时，也应描述传递给过程的参数。被过程改变的函数返回值和全局变量，特别是通过引用参数的那些，也必须在每个过程的起始处描述它们。
### 模块头部注释规范
以一个物理文件为单元的都需要有模块头部注释规范，例如：C#中的.cs文件
用于每个模块开头的说明，主要包括：
1. 功能描述(Description)：   此模块的功能描述与大概流程说明
2. 作者(Author)：
3. 日期(Create Date)：

示例如下：
```csharp
/// <summary>
/// 功能：
/// 完成日期：
/// 作者：
/// </summary>
```

### 方法注释规范
1. C# 提供一种机制，使程序员可以使用含有XML 文本的特殊注释语法为他们的代码编写文档。在源代码文件中，具有某种格式的注释可用于指导某个工具根据这些注释和它们后面的源代码元素生成XML。具体应用当中，类、接口、属性、方法必须有`<Summary>`节，另外方法如果有参数及返回值，则必须有`<Param>`及`<Returns>`节。
2. 事件不需要头注解，但包含复杂处理时（如：循环/数据库操作/复杂逻辑等），应分割成单一处理函数，事件再调用函数。
3. 所有的方法必须在其定义前增加方法注释。
4. 方法注释采用 /// 形式自动产生XML标签格式的注释，列出函数的目的/功能、输入参数、返回值等。
5. 注释使用中文，对于特殊要求的可以使用英文注释
6. **⚠️ 默认跳过**:公共类中，添加方法时必须要添加作者、功能等注释。(这是 .NET Framework 时代遗留规范，现代项目几乎不写 author/功能 注释。**audit 时不主动报这条**，除非用户明确说"按 author 注释规范查一下")
7. **⚠️ 默认跳过**:修改任何方法，必须要添加修改记录的注释。(同上，git blame 已经做了这件事，注释里写修改记录是冗余。**audit 不主动报**)
8. 保证代码和注释的一致性。修改代码同时修改相应的注释，不再有用的注释要删除。
9. 修改bug所添加的代码要添加注释，因为这是一般容易忽略的逻辑，添加注释会对其他人在进行优化或修改代码时，避免再次重新引发bug
10. `<summary>` 标签和注释内容不能写在同一行 —— `<summary>` 和 `</summary>` 必须单独占行，注释内容夹在中间独占一行。其他 XML 文档标签（`<param>`、`<returns>` 等）如果内容很短可以单行写，但 `<summary>` 永远多行。

    正确写法：
    ```csharp
    /// <summary>
    /// 处理订单付款请求
    /// </summary>
    public void ProcessPayment() { }
    ```

    错误写法：
    ```csharp
    /// <summary>处理订单付款请求</summary>
    public void ProcessPayment() { }
    ```

### 代码行注释规范
1.  如果处理某一个功能需要很多行代码实现，并且有很多逻辑结构块，类似此种代码应该在代码开始前添加注释，说明此块代码的处理思路及注意事项等
2. 注释从新行增加，与代码开始处左对齐　　
3. 双斜线与注释之间以空格分开

# 表达式与语句
1.  if、else、else if、for、while、do等语句自占一行，执行语句不得紧跟其后。不论执行语句有多少都要加 { }。
2.  在表达式中使用括号，使表达式的运算顺序更清晰。由于将运算符的优先级与结合律熟记是比较困难的，为了防止产生歧义并提高可读性，即使不加括号时运算顺序不会改变，也应当用括号确定表达式的操作顺序。
3. 避免表达式中的附加功能，不要编写太复杂的复合表达式，适当拆分逻辑益于阅读和维护
4. 不可将浮点变量用 == 或 != 与任何数字比较。
5. 不可在for 循环体内修改循环变量，防止for 循环失去控制。
6. 禁止使用匿名委托，全部换成具名函数。

# 代码布局

## 类内成员顺序

类内部成员**必须**按以下顺序从上到下排列：

1. **私有字段**（含 `private`/`protected` 字段，含 `const` / `static readonly`）
2. **属性**（含 public/private/protected/internal）
3. **事件**（`event` 声明）
4. **构造函数**（含析构函数、静态构造函数）
5. **公有方法**（`public`，含 `internal`）
6. **私有方法**（`protected` 和 `private`）

## region 标记

每个**含两个或以上成员**的分组用 `#region [Name]` / `#endregion` 包裹，名称固定：

- `#region [Private Fields]`
- `#region [Properties]`
- `#region [Events]`
- `#region [Constructors]`
- `#region [Public Methods]`
- `#region [Private Methods]`

**两种免 region 标记的情况**（region 的意义是"分隔不同类型的成员",分隔目标不存在时强加 region 反而是噪声）：

1. **单项分组免标记**：某个分组里只有 1 个成员时，可以省略 region 包裹。
2. **全类只有一种分组也免标记**：整个类**只有一种**非空分组（比如全是 public methods、其他类型都没有），即使该组有多个成员，也可以不加 region —— 因为没有别的组要分隔。

**两种情况下顺序仍然要保持**（多个分组并存时，从上到下:私有字段 → 属性 → 事件 → 构造 → 公有方法 → 私有方法）。

**属性的 backing field 放在 `[Properties]` region 内，紧贴它的属性**（**不放进 `[Private Fields]` region**）：

```csharp
#region [Properties]

private int _no;
public int No
{
    get { return _no; }
    set { _no = value; }
}

private string _name;
public string Name
{
    get { return _name; }
    set { _name = value; }
}

#endregion
```

这种 backing field：
- **不算** `[Private Fields]` 的成员（不会触发 `[Private Fields]` 该不该加 region 的判断）
- **不破坏成员顺序**（在 Properties 区里不算 interleave）
- **和它对应的属性之间不强制空行**（整对算一个单元）
- **两对之间需要空行**（上例 `No` 和 `_name` 之间空一行，因为它们属于不同的属性单元）

## 间隔规则

- **region 之间** —— `#endregion` 和下一个 `#region` 之间必须**空一行**
- **同一 region 内的方法之间** —— 每两个方法之间必须**空一行**
- **同一 region 内的属性之间** —— 每两个属性之间必须**空一行**
- 字段之间不强制空行（短字段允许连续；长字段或带注释的字段建议空行）

## 完整示例

```csharp
public class OrderService
{
    #region [Private Fields]

    private readonly ILogger _logger;
    private readonly IOrderRepository _repository;

    #endregion

    #region [Properties]

    public int PendingCount { get; private set; }

    public bool IsBusy { get; private set; }

    #endregion

    #region [Constructors]

    public OrderService(ILogger logger, IOrderRepository repository)
    {
        _logger = logger;
        _repository = repository;
    }

    #endregion

    #region [Public Methods]

    public void Submit(Order order)
    {
        ValidateOrder(order);
        _repository.Save(order);
    }

    public void Cancel(int orderId)
    {
        var order = _repository.Find(orderId);
        order.Cancel();
    }

    #endregion

    #region [Private Methods]

    private void ValidateOrder(Order order)
    {
        // ...
    }

    private void LogChange(Order order, string action)
    {
        // ...
    }

    #endregion
}
```

## 其他

1. 针对不同没有关联性的功能，可以在 `[Public Methods]` 或 `[Private Methods]` 内部进一步用 `#region [子功能名]` 分类，让功能方法区分明显可读性强
2. 有调用关系的方法、重载的方法等相近或相关的方法尽可能放在一起，方便阅读

# 特殊事项
禁止将枚举类型，使用ToString()转型后拿来赋值或者判断
举例：
```csharp
enum method
{
	GET,
	POST,
}
httpRequest.Method=method.Get.ToString();
```
这种做法会导致代码混淆后，httpRequest.Method实际上="A"或者"a" ，导致http接口请求失败。
正确做法(通过Description注解来获取需要的字符串)：
```csharp
enum method 
{
	[Description("GET")]
	GET, 
	[Description("POST")]
	POST,
 }
 
httpRequest.Method=method.Get.GetDescription();
```
