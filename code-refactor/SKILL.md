# Code Refactor

帮用户改进现有代码的**结构、可读性、效率**,同时保持外部行为不变。

核心是三件事:**先看再动(读引用、读测试、查隐式依赖)**;**先列再改(出重构清单等用户挑要做哪几条)**;**小步推进(一次一件事 + 改完验证)**。

## 工作流概览

下面 0-5 节分别是:**判定 → 读代码勘察 → 列清单 → 实施 → 验证 → 总结**。

**section 2(列清单)绝对不能跳到直接动手** —— 一个看起来无害的"提取函数",这个函数可能在别处被反射调用、或它的名字在测试断言里被字符串匹配。先列再改是为了让用户(以及你自己)对动什么、不动什么有清楚预期。

## 0. 先判定这次请求是不是该由 code-refactor 处理

skill description 已经划过"不适用"边界,被触发时第一件事先确认这次确实是**保持行为不变的重构**:

- **加功能 / 改逻辑** → 不是 refactor。"在登录流程里加个验证码" 是新功能,直接做或走对应流程
- **修 bug** → 不是 refactor。bug fix 会改行为(从"错的"变成"对的"),refactor 是行为不变。**但**用户经常说"重构这段代码顺便修个 bug",这时**分两步**:先把 bug 标出来单独修(独立改动单元,可单独 commit),再做剩下的纯重构
- **语言/框架迁移** → 不是 refactor。"把这个 React 组件改成 Vue"、"Python 改 Go" 是迁移,行为是否一致需要逐条核对,**不要按 refactor 流程做**
- **纯 review,只看不改** → 用 review skill,refactor skill 默认要动代码
- **纯格式化** → 让 prettier / black / gofmt 干,这事 refactor skill 不做
- **加测试 / 写文档** → 这些是新增工作,不是重构现有代码

如果用户实际想做以上任何一种,**主动告诉用户**:"看起来你说的是 X,refactor skill 是'保持行为不变地改进结构',你要做的事行为可能会变。你是想:(a) 我按 X 的思路帮你做 (b) 先把行为变更部分拆出来单独处理,再做剩下的纯重构 (c) 你重新描述一下需求?"

**混合请求**(如"重构这段代码,顺手把 XX 改成 YY")—— **先问清"XX 改成 YY"是否会改行为**,会改的部分隔离开,**不混进 refactor 改动单元里**。混进去的下场是:diff 里既有重命名又有逻辑变更,review 时分不清哪是哪,出了 bug 也不知道是哪步引入的。

除了"行为是否变化"这条红线之外,**默认信任用户给的改动方向**。用户说"改成 X" → 按 X 改,不要反过来挑战"X 是不是好方向" —— 除非 X 明显改行为或破坏依赖(那就走上面的判定流程)。

## 1. 读代码 + 勘察影响范围

**永远先读再动**。即使用户贴了一段代码并说"就改这个",你也至少要确认:

- 这段代码在哪个文件里?用 Glob / Grep 定位
- 这段代码被谁调用?用 Grep 找引用(函数名、类名、导出标识符)
- 有没有测试覆盖?用 Glob 找同名 `*test*` / `*spec*`、或在测试目录 Grep
- 改它会不会影响**字符串反射、序列化字段、配置键名、外部 API**?这些是命名时最容易漏的地方

### 影响范围的"勘察清单"

读代码时**主动找以下几类隐式依赖**(它们不会出现在普通的引用搜索里,但改名/改签名会炸):

- **反射调用**: `getattr(obj, "fieldName")`、`obj["fieldName"]`、Java `getMethod("xxx")`、`eval(...)`、`new Function(...)`、JS `obj[varName]`
- **序列化字段**: JSON / YAML / Protobuf 字段名。改 Python class 属性名,`json.dumps(obj)` 输出的 key 就跟着变了 —— 下游解析方可能炸
- **数据库列名 / 字段名**: ORM 字段映射到 DB 列,改属性名要同步改迁移脚本
- **路由 / URL / handler 名**: HTTP route handler 改名后,路由表 / 装饰器 / 客户端调用路径是否同步?
- **测试断言里的字符串**: `assertEquals("expected", obj.toString())`,改名可能让 toString 输出变化
- **日志 / 监控指标名**: 改函数名可能让监控 dashboard / 报警规则找不到对应指标
- **配置文件里的引用**: YAML/JSON 配置里写着类名/函数名做 dynamic loading

发现以上任一,**在重构清单里明确标"需要同步改 X / Y / Z"**,不要默认"代码改了就行"。

### 如果项目有测试套件,实施前先跑一次基线

读完代码后,在重构清单实施前**先跑一次完整测试**(`npm test` / `pytest` / `go test` / `make test` ...)记录基线:

- 全绿 → 接下来每步改完都要重跑,任何变红立即停下
- 有红 / skip → **告诉用户哪些测试预先就红了**,免得改完被误判成是你弄坏的
- 没有测试 → **明确告诉用户**:"项目里没找到测试套件,这次重构改完无法自动验证,需要你手动确认行为不变。我们要不要继续?需不需要先加几个关键路径的 smoke test?"

## 2. 列重构清单(plan first,不要直接动手)

读完代码、勘察完影响范围之后,把**所有看到的重构机会**列成清单,**让用户从中挑要做哪几条**。即使用户只说了"重构一下",你也要先列清单 → 等用户挑 → 再动手,**不要默认"全做"**。

### 清单格式

用 markdown 表格,每条至少包含:**编号、重构项简述、类型、影响范围、风险、备注**。

```markdown
| # | 重构项 | 类型 | 影响范围 | 风险 | 备注 |
|---|--------|------|---------|------|------|
| 1 | 提取行 23-31 的邮箱校验 → 新函数 `validateEmail` | 局部清理 | 单文件,3 处调用 | 低 | 命名歧义,改完更直观 |
| 2 | 把 `UserService` 的支付逻辑拆到 `PaymentService` | 结构调整 | 3 文件,7 处调用 | 中 | 测试 `UserService.test.ts` 要拆;**改前确认是否要保留旧 API 委托** |
| 3 | 删 `legacyHelper.ts` 整文件 | 死代码 | 0 处引用(已 grep) | 低 | git 历史保底,要删的话直接删 |
```

### 风险等级

- **低**: 单文件内,无外部 API 改动,语义保持。可以连续做几条
- **中**: 跨文件 / 改函数签名 / 改公开类型,影响范围明确,需要同步改调用方。**建议一条一条做,每条改完跑测试**
- **高**: 改公开 API、改字段名(可能影响序列化/DB)、改算法但有 corner case 风险。**强烈建议一条做完充分验证后再下一条**;如果不确定就**先不做**,告诉用户为什么有顾虑

### 类型标签

- **局部清理**: rename / extract / dedupe / simplify / delete-dead
- **结构调整**: split-file / extract-class / change-signature / introduce-abstraction
- **代码异味**: long-function / god-class / long-parameter-list / feature-envy / mystery-naming / primitive-obsession
- **算法优化**: complexity-reduction / batching / caching / lazy-evaluation

### 关于"为了未来"做抽象的克制

清单里**不要塞"为了将来可能要支持 XX 所以抽出 interface"这种抽象**。除非用户明确说要,refactor 的本职是**消除已有的问题**,而不是预测未来。

判断标准: **Rule of Three** —— 真的有 3 处相似才考虑抽象;只有 2 处不要碰(可能只是巧合相似)。如果你看到"这个类只有一个实现但被写成 interface" 反而要标出来,问用户是否要拍平 —— 单实现的 interface 是负债不是资产。

### 列完后说什么

清单列出后,告诉用户:

> 这是我看到的 N 条重构机会。你想做哪几条?也可以直接说"做 1-3,4 跳过"。**如果选中的里面有改公开 API / 字段名 / 大调结构的,在动手前我会再次和你确认**。

**不要在用户没回答前直接动手**。冷启动场景下用户可能希望你帮他挑,这时给出"我的建议:先做风险=低的 1、4、5,中风险的 2 单独讨论,3 等有 perf 用例后再做",**但仍然等用户拍板再动**。

## 3. 实施(逐项,每项一个原子改动)

得到用户确认后,**一项一项做**,不要把多项合在一个改动里。

### 实施一项的标准流程

1. **(改前)再确认一次影响范围**: 如果上一步勘察距离现在已经过了几轮对话,改之前重新 Grep 一次引用 —— 文件可能在期间有变化
2. **改动**: 用 Edit / Write 改对应文件。多文件需要同步改时,**在同一轮工具调用里全改完**(改名/改签名跨多文件时,中间状态会让编译/运行失败)
3. **(改后)验证**:
   - 跑测试 (`npm test` / `pytest` / `go test` ...) → 全绿
   - 类型检查 (`tsc --noEmit` / `mypy` / `go vet`) → 通过
   - 改名时,**再 Grep 一遍旧名**确认无遗漏(包括反射、字符串、注释里有意义的引用)
4. **告诉用户改了什么**: 1-3 句话(改了哪些文件、改了什么、测试结果)

### 改名(最容易出错的操作)

改名漏一处就炸。流程:**Grep 旧名数引用 → 批量替换(单文件 `replace_all`,多文件分批 Edit) → 再 Grep 旧名应当 0 命中(有命中要逐条 review,可能是合法的字符串/注释) → 跑测试**。

**改名是触发"勘察清单"全检查的典型场景** —— 反射、序列化字段、DB 列、配置键名、监控指标都要过一遍。

### 提取函数

圈定代码块 → 找出它用到的外部状态(入参 + 闭包 + 副作用) → 起新名(**描述"做什么"而不是"哪里用",`validateUserAge` 好过 `validateInput`**) → 放在**调用方附近**(被多文件需要才提到公共模块) → 跑测试。

### "提取函数"反例 —— 不要为了"看起来更整洁"切碎逻辑

❌ 不要把一个 30 行的、内聚的逻辑切成 6 个 5 行的私有函数:

```python
def process_order(order):
    _check_inventory(order)
    _calculate_price(order)
    _apply_discount(order)
    _process_payment(order)
    _send_confirmation(order)
    _update_inventory(order)
```

如果每个 `_xxx` 只在这里被调用一次、彼此之间共享 `order` 状态、读者需要在多个函数间跳来跳去才能理解 → **这种"提取"反而降低可读性**。

提取的好处是**消除重复**或**给一段复杂逻辑命名**。如果两者都达不到(只调用一次 + 不需要单独命名),不要做。

### 多文件改动

跨文件改动(改签名、改类型、移动函数)**所有改动放在同一轮工具调用里完成**,不要让中间态出现 —— 编译/测试会炸,debug 时分不清是中间态还是改坏。改动太大时(一个 enum 影响 50 文件),走"**新名 + 旧名并存 → 调用方逐个迁移 → 删旧名**"两步走方案,保中间态可编译。

## 4. 验证 —— 重构的红线是行为不变

每改完一项,**强制做行为验证**。这是 refactor 区别于"瞎改代码"的核心。

"observable behavior"包括: 函数返回值、抛出的异常类型、副作用(IO / 日志 / 监控 / 外部调用)、性能(用户能感知的程度上)、序列化输出、API 响应。这些任一变了都算行为变更,不算纯重构。

### 验证手段(按可信度从高到低)

1. **测试套件全绿**: 改前绿 → 改后还绿 → 强信号"行为不变"
2. **类型检查 / lint 通过**: 至少证明没引入语法 / 类型层面的破坏
3. **运行核心场景**: 没测试但项目能跑,手动跑一遍核心用例
4. **diff 自审**: 上面都没有的情况下,把改动 diff 输出给用户,**逐行解释"这一处改动为什么不改行为"**(等价变换的论证)

### 如果测试变红了

- **立即停下**,不要试图"再多改一点也许就好了"
- 把变红的测试名 + 报错告诉用户
- 选项: (a) 这一项改动撤回 (b) 这个测试的 expectation 需要更新(说明 refactor 不小心改了行为,讨论是否可以接受) (c) 测试本来就是写错的(罕见,需要证据)
- 不要默默把测试断言改成跟新行为一致 —— 这是把"行为变了"伪装成"测试错了"

### 性能优化类的特殊验证

算法 / 性能类的 refactor,除了功能测试不变,还要看**性能确实变好**:

- 有 perf 测试 / benchmark → 跑前后对比,记录数据
- 没有 → **告诉用户**:"我把 O(n²) 改成 O(n) 了,但项目里没有性能用例验证。要不要我加一个最简单的,或者你手动测?" **不要默默说"优化完成"就走了**

## 5. 写完之后告诉用户什么

简洁总结 5 件事:**完成了清单第几条;改了哪些文件;验证结果(测试 / 类型检查 / lint);跳过和不建议做的(各说一句);下次还能接着做什么**。

## 四类重构的具体指引

工作流是统一的,但每一类的常见陷阱不同。

### 局部清理 (local cleanup)

**重命名**:
- 起名标准: 名字描述"是什么 / 做什么",不是"哪里来 / 给谁用"(`validateEmail` 好过 `userLoginEmailCheck`)
- 长度: 局部变量短点 OK(`i`、`acc`、`x`),跨文件可见的标识符要"读了就懂"
- 缩写: 通用缩写 OK(URL、ID、ctx、req、res);自创缩写不要用

**提取函数 / 变量**:
- 触发条件: (a) 一段逻辑重复出现 (b) 一段复杂表达式需要命名解释 (c) 一个 if 条件复杂到读不懂(提取成 `bool` 变量并命名)
- **不要**为了"行数少"而提取(参见上文反例)
- 提取出的函数,**入参列表 ≤ 4 个**;再多说明数据本来就该是一个对象

**消除重复**:
- **Rule of Three**: 出现 3 次再抽;2 次可能只是巧合
- "看起来相似"不等于"逻辑上相同" —— 列出每处的差异,确认抽出来的版本能完全覆盖
- 抽出来的公共函数放哪儿: 取**最低共同祖先目录**(三处分别在 `services/`、`utils/`、`api/`,公共函数放 `utils/`)

**简化分支**:
- 早返回 / guard clauses 减少嵌套: `if (!x) return; ...` 好过 `if (x) { 大段逻辑 } else { return }`
- 多个 if-else 链 → 看能不能换查表 / 多态(但多态有引入抽象成本,慎用)
- **嵌套的三元运算符** → **永远改成 if-else 或拆变量**;三元嵌套是可读性杀手
- 双重否定 → 改单重肯定(`if (!isNotEmpty)` → `if (isEmpty)`)

**删死代码**:
- Grep 确认 **0 处引用**(包括动态字符串、反射 —— 见勘察清单)
- 注释掉的代码块 → **直接删**,git 历史还在(代码不是版本管理)
- 删整文件: 先 Grep 文件名,再 Grep 文件里所有 export 标识符
- 配置项 / 环境变量 → 还要检查部署脚本、CI 配置、文档里有没有提及

### 结构调整 (structural refactoring)

**拆分文件**:
- 触发: 单文件 > ~500 行,且能找到清晰的内聚边界
- 拆分点: 看 import 图 + 找"哪些函数互相调用,哪些不调用"
- 拆完检查: 是否引入了循环依赖

**提取类 / 模块**:
- 触发: 一组函数操作同一组数据 / 一个类承担了 > 1 个职责(职责描述要用"和")
- 提取后**保留原 API**(原类委托给新类的方法),让调用方迁移更平滑;不要强求一次断舍离

**调整签名**:
- **永远先 Grep 所有调用点**,数清楚要改多少处
- 多文件改动: 在同一轮 Edit 里全改完
- 加可选参数(default 值)→ 低风险
- **改必填参数顺序 / 类型 / 名字**→ 高风险,优先用"加新参数 + deprecate 旧的"两步走

**引入抽象 (interface / 基类)**:
- 触发: 真的有 ≥ 3 个实现,且它们的差异不是细节而是"做同一件事的不同方式"
- 反模式: 只有 1 个实现的 interface;**这种情况下应该删 interface**

### 代码异味识别 (smell detection)

当用户说"扫一下代码异味"或类似,**主动**扫以下几类。每发现一处,列在清单里(标 `代码异味` 类型),让用户挑要不要修。

#### 常见异味清单

| 异味 | 怎么发现 | 怎么修 |
|------|---------|--------|
| Long function | 单函数 > ~50 行,或圈复杂度 > ~10 | 提取子函数(参考上文反例,不要为提取而提取) |
| God class / God module | 单文件 > 500 行,职责描述要用"和" | 拆模块 / 提取类 |
| 长参数列表 | 函数参数 > 4 个 | 参数对象化、或函数本来就该拆 |
| 重复代码 | grep 看 / 眼看 | Rule of Three 后提取 |
| Feature envy | A 类的方法大量访问 B 类的字段 | 把方法移到 B 类,或反过来想 A/B 是不是该合并 |
| Primitive obsession | 到处用 string / int 表达本应是类型的概念(UserId 用 string) | 引入类型别名 / 包装类 |
| 神秘命名 | `data`、`info`、`temp`、`x1`、`mgr` 这种 | 重命名 |
| 注释解释代码 | 代码里有"这段做了 X,因为 ..." 的注释 | 通常说明命名 / 拆分有问题。修代码,让注释不必要 |
| 死代码 | grep 无引用 | 删 |
| 神秘数字 / 字符串 | `if (status == 3)` | 提取命名常量 |

#### 异味识别 ≠ 必须修

异味是**信号,不是规则**。50 行的函数如果做的是一件内聚的事,可能比拆成 6 个 8 行函数更可读。**列出来让用户判断**,不要默认"扫出来的都要修"。

特别地:**用户没让你扫异味时,不要主动扫整个项目**。skill 的触发是"用户主动指了某段/某文件",扫异味也限定在那个范围内。

### 性能 / 算法优化

这一类**最容易在 refactor 名义下改行为**,要格外小心。

#### 优化前必须确认

- **真的有性能问题吗?** 有 profile 数据 / 用户反馈慢?还是只是"看起来 O(n²) 不爽"?**没有数据证明慢的代码,不要为了"理论上更快"动手**
- **优化后行为是否完全一致?** 有些"优化"会偷偷改语义,逐项检查下表

#### 常见优化模式 + 它们的"行为陷阱"

| 模式 | 收益 | 行为陷阱(改前必须想清楚) |
|------|------|---------|
| O(n²) 嵌套循环 → hash map O(n) | 大数据集质变 | hash map 不保序;原循环可能依赖顺序 |
| 批量化 DB 查询(N+1 → 1) | 网络往返减少 | 事务边界变化;一个失败原来跳过,现在可能全 fail |
| 加 memoize / cache | 重复计算消除 | 缓存失效策略错了就读到旧数据;副作用只发生一次了(原来每次都发生) |
| List → Set / Dict | O(n) 查 → O(1) 查 | 顺序丢失;重复元素被合并 |
| Lazy / generator | 大数据集省内存 | 副作用时机变;迭代器只能消费一次;异常被推迟 |
| 并行 / 异步化 | 总时长缩短 | 结果顺序不定;副作用顺序不定;错误处理边界变 |

每做一种,在清单里**显式列出对应行为陷阱**,让用户判断是否可接受。**陷阱是真实存在的可能性,不是"可能没事"**。

#### 优化后必须验证

参见 [Section 4 → 性能优化类的特殊验证](#性能优化类的特殊验证):跑前后 benchmark + 跑功能测试;**特别注意**:Big-O 改善不等于实测变快(常数因子可能更大,小数据集反而变慢)。
